Version control with Git and GitHub

What is version control?

Imagine you are writing a report. You save it as report.docx. Then you make changes and save it as report_v2.docx. Then report_final.docx. Then report_final_REALLY_final.docx. Sound familiar?

Now imagine that with code. You have a folder full of files, you make a change that breaks everything, and you desperately try to remember what it looked like before. Or a colleague wants to work on the same code at the same time and you have to email files back and forth, merging changes by hand.

A Version Control System (VCS) solves all of this. It is a tool that tracks every change you make to your files over time. You can think of it as a very detailed “undo history” for your entire project, one that never forgets, even after you close your editor.

With a VCS you can:

  • Go back in time — restore any previous version of any file.
  • See what changed — compare two versions and see exactly which lines were added, removed, or modified.
  • Understand why it changed — every saved version has a message explaining what was done and who did it.
  • Collaborate safely — multiple people can work on the same project simultaneously without overwriting each other’s work.
  • Experiment without fear — try a risky idea knowing you can always undo it completely.

Git is by far the most popular VCS in the world. It is used by millions of developers, from solo hobbyists to the teams that build Linux, Python, and virtually every major software project you can think of.

Installing Git

Follow the git install instructions from the official git page.

Telling Git who you are

Before using git for the first time, you need to tell it your name and email. Git records this information alongside every change you make, so collaborators know who did what.

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

You only need to do this once. You can verify it worked:

git config --global user.name
git config --global user.email

The project we will build

Throughout this tutorial we will build a small Python utility called converter.py — a script with a couple of unit conversion functions. It is intentionally simple: the goal is to learn git, not Python.

By the end, our project folder will contain:

unit-converter/
├── converter.py
└── README.md

Creating a repository

A repository (or “repo”) is a project folder that git is tracking. To create one, navigate to the folder you want to use and run git init.

Let’s start from scratch:

# Create a new folder and enter it
mkdir unit-converter
cd unit-converter

# Initialise a git repository inside it
git init

You should see something like:

Initialized empty Git repository in /home/yourname/unit-converter/.git/

Git created a hidden folder called .git/ inside your project. That folder is where git stores all its history. You should never need to touch it manually, just know it’s there.

Checking the status of your repository

git status is one of the most useful commands. It tells you the current state of your repository: which files have changed, which are new, which are ready to be saved.

git status

Right now the repository is empty, so you will see:

On branch main

No commits yet

nothing to commit (create/copy files and start tracking)

Creating our first file

Let’s create the README file. A README.md is a Markdown file that describes the project. It is the first thing people read when they find your project.

Create a file called README.md in the unit-converter/ folder with this content:

# Unit Converter

A small Python utility to convert between common units.

## Functions

- `km_to_miles(km)` — converts kilometres to miles.
- `celsius_to_fahrenheit(c)` — converts Celsius to Fahrenheit.

## Usage

```python
from converter import km_to_miles, celsius_to_fahrenheit

print(km_to_miles(10))        # 6.21
print(celsius_to_fahrenheit(100))  # 212.0
```

Now run git status again:

git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md

nothing added to commit but untracked files present

Git has noticed the new file, but it says it is untracked. Git does not automatically record every file, you have to explicitly tell it which ones to watch.

The three areas of Git

Before going further, it helps to understand how git thinks about files. There are three areas:

Working directory  →  Staging area  →  Repository (history)
   (your files)        (git add)         (git commit)
  1. Working directory: the actual files on your computer. This is where you do your work.
  2. Staging area (also called the “index”): a waiting room. You decide which changed files to include in the next saved snapshot.
  3. Repository: the permanent history. Once a snapshot is committed here, it is recorded forever.

This two-step process (stage, then commit) might seem unnecessary at first, but it gives you fine-grained control over what goes into each saved snapshot.

Staging files with git add

To tell git to start tracking a file (and to stage it for the next commit), use git add:

git add README.md

Run git status again:

git status
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   README.md

The file has moved from “untracked” to “changes to be committed”. It is in the staging area, ready to be committed.

Saving a snapshot with git commit

A commit is a permanent snapshot of the staged changes, saved into the repository history. Every commit has:

  • A message: a short description of what changed and why.
  • A timestamp: when it was created.
  • An author: who made it.
  • A unique ID: a long string of letters and numbers (called a hash) that uniquely identifies this commit.
git commit -m "Add README with project description"
[main (root-commit) a3f9c12] Add README with project description
 1 file changed, 15 insertions(+)
 create mode 100644 README.md

You have just made your first commit. The history of this project now has one entry.

TipWriting good commit messages

A commit message should complete the sentence: “If applied, this commit will…”

  • ✅ “Add README with project description”
  • ✅ “Fix bug in celsius_to_fahrenheit for negative values”
  • ❌ “changes”
  • ❌ “asdfgh”
  • ❌ “fix stuff”

Clear messages make the history readable — especially when you are trying to understand what happened six months ago.

Adding the Python code

Now let’s create converter.py:

# converter.py

def km_to_miles(km):
    """Convert kilometres to miles."""
    return km * 0.621371

def celsius_to_fahrenheit(celsius):
    """Convert Celsius degrees to Fahrenheit."""
    return celsius * 9 / 5 + 32

Stage it and commit:

git add converter.py
git commit -m "Add converter.py with km_to_miles and celsius_to_fahrenheit"

We now have two commits. Let’s look at the history.

Viewing the history with git log

git log shows the list of all commits, most recent first:

git log

    Add converter.py with km_to_miles and celsius_to_fahrenheit

Each commit shows:

  • The hash — the long unique ID (you usually only need the first 7 characters).
  • The author and date.
  • The commit message.

For a more compact view, use:

git log --oneline
7b2e841 Add converter.py with km_to_miles and celsius_to_fahrenheit

This is the view you will use most often.

Making changes and seeing the difference

Let’s improve our converter by adding a new function. Open converter.py and add a third function at the end:

# converter.py

def km_to_miles(km):
    """Convert kilometres to miles."""
    return km * 0.621371

def celsius_to_fahrenheit(celsius):
    """Convert Celsius degrees to Fahrenheit."""
    return celsius * 9 / 5 + 32

def kg_to_pounds(kg):
    """Convert kilograms to pounds."""
    return kg * 2.20462

Before staging the change, let’s use git diff to see exactly what we modified:

git diff
diff --git a/converter.py b/converter.py
index 4b3c1a2..9f7d3e8 100644
--- a/converter.py
+++ b/converter.py
@@ -7,3 +7,7 @@ def celsius_to_fahrenheit(celsius):
     """Convert Celsius degrees to Fahrenheit."""
     return celsius * 9 / 5 + 32
+
+def kg_to_pounds(kg):
+    """Convert kilograms to pounds."""
+    return kg * 2.20462

Lines starting with + are additions, lines with - are deletions. This is exactly what has changed since the last commit.

Now stage and commit:

git add converter.py
git commit -m "Add kg_to_pounds function to converter"

Also update the README to mention the new function:

## Functions

- `km_to_miles(km)` — converts kilometres to miles.
- `celsius_to_fahrenheit(c)` — converts Celsius to Fahrenheit.
- `kg_to_pounds(kg)` — converts kilograms to pounds.
git add README.md
git commit -m "Update README to document kg_to_pounds"

Our log now has four commits:

git log --oneline
d4a1f93 Update README to document kg_to_pounds
c8e2b74 Add kg_to_pounds function to converter
7b2e841 Add converter.py with km_to_miles and celsius_to_fahrenheit
a3f9c12 Add README with project description

Staging all modified files at once

When you have modified several files and want to stage all of them, you can use:

git add .

The . means “everything in the current folder”. This is convenient, but use it carefully — you might accidentally stage files you did not mean to include.

Going back in time

One of the most powerful features of git is the ability to look at, or even restore, any previous state of your project.

Viewing an old version

To inspect what a file looked like at a particular commit, use git show followed by the commit hash and the file name:

git show a3f9c12:README.md

This prints the content of README.md as it was in that first commit, without changing anything in your working directory.

Comparing two commits

To see what changed between two commits, use git diff with two hashes:

git diff a3f9c12 d4a1f93

This shows every change that happened between the first commit and the latest one.

Temporarily travelling to an old commit

To actually switch your entire project back to a previous state, use git switch --detach with a commit hash:

git switch --detach 7b2e841
HEAD is now at 7b2e841 Add converter.py with km_to_miles and celsius_to_fahrenheit

Git will tell you that HEAD is detached. This just means you are looking at an old snapshot. You can open and read files, they will all look as they did at that commit. To go back to the present (the latest commit):

git switch main

Everything is restored to the latest state. Nothing was lost.

Undoing changes to a file

If you have made changes to a file that you have not committed yet and you want to discard them and go back to the last committed version:

git restore converter.py
WarningBe careful with restore

git restore <file> permanently discards your uncommitted changes to that file. Unlike most things in git, changes that were never committed cannot be recovered. Use it with care.

Undoing a commit safely with git revert

What if you want to undo a commit that is already in the history? The safest way is git revert. It creates a new commit that undoes the changes from the target commit, without rewriting the history:

# Undo the last commit (but keep the history intact)
git revert HEAD

Git will open a text editor for you to write a commit message (you can accept the default). After saving, a new commit will appear in the log that cancels out the previous one. This approach is safe to use even when collaborating with others.

TipIf git opens an unfamiliar editor

By default, git may open vim, which can be confusing. To save and exit vim, press Esc, then type :wq and press Enter. To avoid the editor entirely, you can use git revert HEAD --no-edit to accept the default message.

You can also configure git to use a friendlier editor, for example VS Code:

git config --global core.editor "code --wait"

What should go in a repository?

Not everything belongs in git. The general rule is: commit source files, not generated files.

Do commit:

  • Your Python source files (.py)
  • Documentation and README files (.md)
  • Configuration files

Do not commit:

  • Generated files (compiled code, build outputs)
  • Large binary files (images, datasets, videos)
  • Files with secrets (passwords, API keys)
  • Editor or OS clutter (__pycache__/, .DS_Store)

The .gitignore file

You can tell git to ignore certain files by creating a file called .gitignore in your project root. Each line is a pattern of files to ignore:

# .gitignore
__pycache__/
*.pyc
.DS_Store
.env

Create this file, then stage and commit it:

git add .gitignore
git commit -m "Add .gitignore to exclude Python cache files"

From now on, git will not suggest staging those files.

Introducing GitHub

So far, everything has happened on your own computer. GitHub is a website that hosts git repositories online. It adds several things on top of git:

  • A remote backup of your repository.
  • A way to share your code with the world (or just your team).
  • Tools for collaboration — colleagues can download your code, make changes, and propose them back to you.
  • A visual interface to browse your history, compare commits, and read your README beautifully rendered.
Note

GitHub is not the only option, GitLab and Bitbucket are alternatives, but it is by far the most widely used.

Creating a GitHub account

Go to https://github.com and sign up for a free account if you do not have one already.

Creating a remote repository on GitHub

  1. Once logged in, click the + button in the top-right corner and choose New repository.
  2. Give it the name unit-converter.
  3. Leave it Public (or choose Private if you prefer).
  4. Do not check “Add a README file” — we already have one locally.
  5. Click Create repository.

GitHub will show you a page with instructions. We want the section called “…or push an existing repository from the command line”.

Connecting your local repo to GitHub

GitHub will show you two commands. Run them in your terminal:

# Tell your local repo where the remote one is (replace with your actual URL)
git remote add origin https://github.com/your-username/unit-converter.git

# Push your commits to GitHub
git push -u origin main

origin is just the nickname git uses to refer to the remote repository. You could call it anything, but origin is the convention everyone uses.

After the push, refresh your GitHub page, you should see your files and your commit history there.

Pushing future commits

After the first push, whenever you make new commits locally and want to send them to GitHub, you only need:

git push

Collaborating with a colleague

Here is how two people can work on the same project.

Your colleague clones the repository

Your colleague goes to your GitHub repository page, copies the URL, and runs:

git clone https://github.com/your-username/unit-converter.git
cd unit-converter

git clone downloads the entire repository — all files and all history — to their computer. They now have an identical copy.

Your colleague makes a change and pushes it

Your colleague adds a new function to converter.py:

def miles_to_km(miles):
    """Convert miles to kilometres."""
    return miles / 0.621371

They save the file, commit, and push:

git add converter.py
git commit -m "Add miles_to_km function"
git push

Their changes are now on GitHub.

You pull the changes

Back on your computer, you run:

git pull

Git downloads the new commits from GitHub and updates your local files. You now have the miles_to_km function too, without having to copy any files manually.

git pull is something you should run at the start of every working session when collaborating, to make sure you have the latest version of the code.

WarningWhat if git push is rejected?

If your colleague pushed changes before you did, your git push will fail with a message saying your branch is behind the remote. This is normal — git is protecting you from overwriting their work. Simply run git pull first to download their changes, and then git push again.

Most of the time git pull will merge everything automatically. But if you and your colleague both edited the same lines in the same file, git does not know which version to keep. This is called a merge conflict.

When a conflict happens, git will mark the affected lines in the file like this:

<<<<<<< HEAD
your version of the line
=======
your colleague's version of the line
>>>>>>> their-commit-hash

To resolve the conflict, open the file, decide which version to keep (or combine both), remove the <<<<<<<, =======, and >>>>>>> markers, and then stage and commit the result. Most editors, including VS Code, highlight conflicts and offer buttons to accept one side or the other, making the process much easier.

Conflicts are a normal part of collaboration — they are not errors. They simply mean two people touched the same spot and a human needs to decide the final version.

A typical daily workflow

Here is a summary of what a typical working session looks like:

# 1. Start by getting the latest changes from your team
git pull

# 2. Do your work — edit files, add new ones, etc.

# 3. Check what has changed
git status
git diff

# 4. Stage the files you want to commit
git add converter.py
git add README.md
# or: git add . (to stage everything)

# 5. Commit with a clear message
git commit -m "Add litres_to_gallons function"

# 6. Push your work to GitHub
git push

Using git from a graphical interface

Throughout this tutorial we have used the terminal to run git commands. This is useful for understanding what git does under the hood, but in practice most developers use a graphical interface for their day-to-day git work.

VS Code, for example, has excellent built-in git support. The Source Control panel (the branch icon in the left sidebar, or Ctrl+Shift+G) lets you do everything we have covered — staging files, writing commit messages, viewing diffs, pushing, pulling — with just a few clicks. Other editors like PyCharm, RStudio, and Sublime Text offer similar integrations.

There are also standalone git GUIs such as GitHub Desktop or GitKraken, which provide a visual overview of your branches and history.

Whichever tool you choose, the concepts are the same: staging, committing, pushing, pulling. The terminal commands you learned here map directly to the buttons in any GUI, so you will always know what is happening behind the scenes.

  4. Run git status again — secret.txt should no longer appear.

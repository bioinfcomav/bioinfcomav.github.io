Testing your Python code with pytest

Why do we test code?

Imagine you write a function to convert temperatures from Celsius to Fahrenheit. It seems to work when you try a couple of values by hand, so you move on. Two weeks later, a colleague tells you the weather app you both built is showing wrong temperatures in extreme cold. The bug was there all along, you just never checked that edge case.

Tests are small pieces of code that automatically check whether your functions behave as expected. Think of a function as a contract, the function is supposed to carry out a particular task and the test assures you that it is really doing what is supposed to do. They let you:

  • Make sure that your functions are doing what they are supposed to do.
  • Catch bugs early.
  • Safely change or improve your code, knowing the tests will alert you if something breaks.

Good developers write tests alongside their code, not as an afterthought.

Setting up pytest

pytest is a popular Python testing library. It is not part of the standard library, so you need to install it in your project (if you don’t know about uv check out its tutorial):

$ uv add pytest

To verify the installation worked:

$ uv run pytest --version

You should see something like pytest 9.x.x.

Your first test

Let’s start with the simplest possible example: a function that adds two numbers.

def add(a, b):
    return a + b


if __name__ == "__main__":
    # Let's try it manually
    print(add(2, 3))
    print(add(-1, 1))
    print(add(0, 0))

This looks correct, but running it by hand every time we change the code is tedious and error-prone. Let’s write a test instead. Save this code in a file named add.py

In pytest, a test is just a regular Python function whose name starts with test_. Inside it, we use assert to check that something is true.

Here is what the test file would look like. Save it as test_add.py:

from add import add


def test_add_positive_numbers():
    assert add(2, 3) == 5


def test_add_negative_numbers():
    assert add(-1, -1) == -2


def test_add_zero():
    assert add(0, 5) == 5

To run the tests, open a terminal in the same folder and type:

$ uv run pytest test_add.py

You will see output like this:

Each dot (.) represents one test that passed. If a test fails, you get an F and a detailed explanation of what went wrong.

What does assert do?

assert checks that a condition is True. If it is not, it raises an AssertionError and pytest marks that test as failed.

When pytest runs a test and an assert fails, it prints a clear message showing you the expected value versus what your function actually returned.

Falining tests

Let’s create a wrong subtract function and its tests.

def subtract(a, b):
    return a + b
from subtract import subtract


def test_subtract_positive_numbers():
    assert subtract(2, 3) == -1
  1. Run the tests.
  2. Check the failure.
  3. Fix the subtract function.
  4. Run the tests again.

How pytest finds your tests

pytest automatically discovers tests by looking for:

  • Files whose names start with test_ or end with _test.py.
  • Functions inside those files whose names start with test_.

This means you don’t have to register your tests anywhere, just follow the naming convention and run pytest.

Testing unit conversions

Unit conversion functions are great for practising tests because the expected results are mathematically defined and easy to verify.

Let’s write the module temperature.py with the temperature conversion functions:

def celsius_to_fahrenheit(celsius):
    return celsius * 9 / 5 + 32


def fahrenheit_to_celsius(fahrenheit):
    return (fahrenheit - 32) * 5 / 9


if __name__ == "__main__":
    # Manual checks
    print(celsius_to_fahrenheit(0))  # Water freezes: should be 32
    print(celsius_to_fahrenheit(100))  # Water boils: should be 212
    print(fahrenheit_to_celsius(32))  # Should be 0
    print(fahrenheit_to_celsius(212))  # Should be 100

Now let’s write tests. Notice how the test names describe the scenario being tested, this makes failing tests much easier to understand:

from temperature import celsius_to_fahrenheit, fahrenheit_to_celsius


def test_water_freezing_point():
    assert celsius_to_fahrenheit(0) == 32


def test_water_boiling_point():
    assert celsius_to_fahrenheit(100) == 212


def test_body_temperature():
    assert celsius_to_fahrenheit(37) == 98.6


def test_fahrenheit_to_celsius_freezing():
    assert fahrenheit_to_celsius(32) == 0


def test_fahrenheit_to_celsius_boiling():
    assert fahrenheit_to_celsius(212) == 100

To run the tests in a terminal. If you run pytest without any extra arguments it will find all the tests files and it will run all the tests in them:

$ uv run pytest

Comparing floating-point numbers

There is a subtle problem with floating-point arithmetic in computers: the result of a calculation is sometimes not exactly what you expect.

Because of this, comparing floating-point numbers with == can fail even when the values are essentially correct. pytest provides a helper called approx to handle this:

from pytest import approx

def test_body_temperature():
    # pytest.approx allows a tiny margin of error
    assert celsius_to_fahrenheit(37) == approx(98.6)


def test_absolute_zero():
    assert celsius_to_fahrenheit(-273.15) == approx(-459.67)

pytest.approx considers two floats equal if they are very close to each other (by default within a relative tolerance of 1e-6). Always use it when testing functions that return floating-point results.

from conversions import string_to_int, int_to_binary_string, list_to_comma_string

    assert int_to_binary_string(1) == "1"

    assert int_to_binary_string(255) == "11111111"

Notice how we test multiple cases for each function: normal values, edge cases like zero, negative numbers, and single items. A good test suite thinks about all the situations a function might encounter.

Testing that a function raises an exception

Sometimes a function is supposed to raise an exception when it receives invalid input. For example, string_to_int("hello") cannot convert the word “hello” to a number, so Python raises a ValueError.

Testing that the right exception is raised is just as important as testing that the function returns the right value.

pytest provides pytest.raises to test for exceptions. Add the following tests to test_conversions.py:

# test_conversions.py (extended)
import pytest

def test_string_to_int_invalid_raises_error():
    with pytest.raises(ValueError):
        string_to_int("hello")


def test_string_to_int_empty_raises_error():
    with pytest.raises(ValueError):
        string_to_int("")

The with pytest.raises(ValueError): block tells pytest: “I expect this code to raise a ValueError. If it does, the test passes. If it does not raise the error (or raises a different one), the test fails.”

Let’s see a more complete example with a function that validates its input. Add the following code to temperature.py.

And the corresponding tests to test_temperature.py

from temperature import celsius_to_kelvin
import pytest


def test_water_freezing_in_kelvin():
    assert celsius_to_kelvin(0) == pytest.approx(273.15)


def test_water_boiling_in_kelvin():
    assert celsius_to_kelvin(100) == pytest.approx(373.15)


def test_absolute_zero_in_kelvin():
    assert celsius_to_kelvin(-273.15) == pytest.approx(0.0)


def test_below_absolute_zero_raises_error():
    with pytest.raises(ValueError):
        celsius_to_kelvin(-300)


def test_way_below_absolute_zero_raises_error():
    with pytest.raises(ValueError):
        celsius_to_kelvin(-1000)```

Writing tests alongside your code

A healthy habit is to write tests as you write the code, not after. Here is a typical workflow:

  1. Write a small function.
  2. Write one or two tests for it.
  3. Run pytest. Make sure the tests pass.
  4. Add more tests for edge cases.
  5. Run pytest again.

This way you build confidence in your code as you go, and bugs are caught immediately rather than weeks later.

NoteExercise 6 — Tests first: clamp a number

In this exercise the tests are already written for you. Your task is to implement the function so that all of them pass.

This is how professional developers often work: they write the tests before the code, a practice known as Test-Driven Development (TDD). The workflow is sometimes called red-green:

  1. Red — run the tests and watch them fail (they are red because the function does nothing yet).
  2. Green — write just enough code to make every test pass.

Save the following two files and run uv run pytest test_clamp.py. You will see all tests fail. Then open clamp.py, replace pass with your implementation, and run pytest again until every test is green.

clamp.py:

def clamp(value, minimum, maximum):
    """Return value clamped to the range [minimum, maximum].

    If value is less than minimum, return minimum.
    If value is greater than maximum, return maximum.
    Otherwise, return value unchanged.

    Raises ValueError if minimum is greater than maximum.
    """
    pass

test_clamp.py:

import pytest
from clamp import clamp


def test_value_within_range():
    assert clamp(5, 0, 10) == 5


def test_value_below_minimum():
    assert clamp(-3, 0, 10) == 0


def test_value_above_maximum():
    assert clamp(15, 0, 10) == 10


def test_value_equal_to_minimum():
    assert clamp(0, 0, 10) == 0


def test_value_equal_to_maximum():
    assert clamp(10, 0, 10) == 10


def test_negative_range():
    assert clamp(-5, -10, -1) == -5


def test_clamp_below_negative_range():
    assert clamp(-20, -10, -1) == -10


def test_clamp_above_negative_range():
    assert clamp(0, -10, -1) == -1


def test_minimum_equals_maximum():
    assert clamp(5, 3, 3) == 3


def test_invalid_range_raises_error():
    with pytest.raises(ValueError):
        clamp(5, 10, 0)
